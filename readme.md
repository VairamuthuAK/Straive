**Scrapy Course/Directory/Calendar Framework**
This framework standardizes how spiders are created, configured, and post-processed for course, directory, and calendar data scraping.
It supports both pure Scrapy spiders and multi-tool spiders that combine Scrapy, Python requests, and Playwright in a single workflow.

1. Creating a Scrapy Project

Create and enter a new Scrapy project:
`scrapy startproject myproject`
`cd myproject`

This generates the standard Scrapy structure (scrapy.cfg, package folder, spiders/ directory, etc.).

After project creation:

- Add spider scripts under myproject/spiders/.

- Maintain a requirements.txt file at the project root.

- Run spiders using:
`scrapy crawl <spider_name>`

`scrapy crawl iccms_mixed`

2. Dependency Management (requirements.txt)
Every project must include a requirements.txt file.
When new external libraries are introduced, add them here.

Example:
- scrapy==2.11.0
- pandas==2.2.2
- pandera==0.20.0
- requests==2.32.3
- playwright==1.49.0

Installing dependencies:

`pip install -r requirements.txt`

3. Running a Spider

All spiders now support dynamic modes via the SCRAPE_MODE setting.
This determines which modules (course, directory, calendar) will be scraped.

* scrapy crawl iccms_mixed -s SCRAPE_MODE=all
* scrapy crawl iccms_mixed -s SCRAPE_MODE=course
* scrapy crawl iccms_mixed -s SCRAPE_MODE=directory
* scrapy crawl iccms_mixed -s SCRAPE_MODE=calendar
* scrapy crawl iccms_mixed -s SCRAPE_MODE=calendar_course


Behavior:
* SCRAPE_MODE=all → runs all three (course + directory + calendar)
* SCRAPE_MODE=course → runs only course
* SCRAPE_MODE=directory → runs only directory
* SCRAPE_MODE=calendar → runs only calendar
* SCRAPE_MODE=calendar_course → runs both calendar and course

This modular logic lets developers run one or multiple data types in a single spider, improving reusability and execution control.

4. Spider Templates
Two templates are available:

* Pure Scrapy – standard spider fetching all three components with Scrapy Requests.

* Mixed‑tool spider – integrates:
    - Scrapy for course requests
    - Python requests for API‑based directory calls
    - Playwright (headless Chromium) for dynamic pages like calendars

All templates use unified function names:

- parse_course()
- parse_directory()
- parse_calendar()

Each spider outputs three CSVs, named by convention:

<spider_name>_course.csv
<spider_name>_directory.csv
<spider_name>_calendar.csv

For example:

- iccms_course.csv
- iccms_directory.csv
- iccms_calendar.csv

5. Shared Utilities (utils.py)

`clean_dataframe(df)`
Cleans and normalizes all cell values in a DataFrame:

- Converts non-strings safely to strings.
- Fixes encoding issues (latin-1 → utf-8).
- Removes newlines and excessive whitespace.
- Collapses multiple spaces.
- Strips leading/trailing whitespace.

`validate_df(df, kind)`

Validates a DataFrame against a strict Pandera schema:

- kind can be "course", "directory", or "calendar".
- Ensures proper column names and data types.
- strict=True rejects unexpected columns.
- coerce=True casts values to proper types.
- Raises SchemaError or ValueError for invalid frames.

6. Post‑Processing in Pipelines
The item pipeline performs cleaning and validation based on the current SCRAPE_MODE.

During `spider_closed()`:

* The pipeline detects which modules were run using the mode.
* Deletes existing _final.csv files that match the mode.
* Loads raw CSVs generated by spiders.
* Cleans and validates them using clean_dataframe() and validate_df().
* Writes new <spider_name>_<module>_final.csv files.

If the spider ran with:
`scrapy crawl iccms_mixed -s SCRAPE_MODE=course_directory`
then only:
- iccms_course.csv
- iccms_directory.csv

will be cleaned, validated, and re-exported as:
- iccms_course_final.csv
- iccms_directory_final.csv
